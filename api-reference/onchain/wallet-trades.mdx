---
title: "Trade History (Wallet)"
description: "Complete onchain trade fill history for any wallet. Every fill, every counterparty, every fee."
openapi: "GET /v2/onchain/wallets/{address}/trades"
---

Returns every onchain trade fill for a wallet, including both maker and taker sides. This is the complete settlement record from the blockchain. Each trade is enriched with market metadata (question, slug, outcome, image).

Unlike the standard [wallet trade history](/api-reference/wallets/trades) which relies on Polymarket's data API (and can timeout or miss trades for heavy wallets), this endpoint pulls directly from onchain settlement data and never misses a fill.

<Tip>
**Need to bulk-export the wallet's entire trade history in one call?** See [Trade History (Wallet) — Bulk](/api-reference/onchain/wallet-trades-all) — Growth plan and above can fetch the full history in one response (no pagination, up to 250K trades).
</Tip>

## Two pagination modes

This endpoint supports both **offset** and **cursor** pagination. They produce identical data; cursor is much faster at depth.

| Mode | Trigger | Speed | When to use |
|---|---|---|---|
| **Offset** (default) | `?offset=N` (or no pagination params) | Slows linearly with depth — fast for the first ~25K results, then degrades | Casual use, small wallets, getting just the first page |
| **Cursor** (recommended for paging) | Pass `?cursor=` (empty for first page, then `?cursor=<lastTs>:<lastId>` from prior response) | **Constant ~1 second per page at any depth** | Walking the entire trade history of a wallet, deep pagination, heavy traders |

**Cursor walks any wallet in seconds-per-page regardless of how many trades it has.** Available on every paid tier — no growth restriction.

For a one-shot bulk export of an entire wallet's trade history in a single response (no pagination needed), see [Trade History (Wallet) — Bulk](/api-reference/onchain/wallet-trades-all) — Growth plan and above.

## Request

```
GET /v2/onchain/wallets/{address}/trades?limit=100&offset=0
GET /v2/onchain/wallets/{address}/trades?limit=1000&cursor=<cursor>
```

| Parameter | Type | Location | Description |
|-----------|------|----------|-------------|
| `address` | string | path | Wallet address (0x-prefixed, 40 hex chars) |
| `limit` | integer | query | Max results (default 100, max 1000) |
| `offset` | integer | query | Offset mode: skip first N results. Used only when `cursor` is absent. Slow at depth (>25K). |
| `cursor` | string | query | Cursor mode: pass empty string `?cursor=` for first page, then echo back `pagination.cursor` from the response for each subsequent page. Format: `<lastTs>:<lastId>`. Constant-time at any depth. |

## `side` vs `direction`

These two fields look similar but answer different questions, and both are populated on every row:

| Axis | Field | What it tells you |
|---|---|---|
| Exchange role | `side` | Did the queried wallet provide liquidity (maker) or take it (taker)? |
| Trader intent | `direction` | Did the queried wallet enter (BUY) or exit (SELL) outcome shares? |

A maker can be either a buyer or a seller depending on which side of the orderbook they posted to. A taker likewise. The two axes are independent.

You will see all four combinations in real wallets:

| `side` | `direction` | Means |
|---|---|---|
| `maker` | `BUY` | Wallet posted a resting buy order, was hit by a taker selling outcome shares |
| `maker` | `SELL` | Wallet posted a resting sell order, was hit by a taker buying outcome shares |
| `taker` | `BUY` | Wallet crossed the spread to buy outcome shares immediately |
| `taker` | `SELL` | Wallet crossed the spread to sell outcome shares immediately |

The information `direction` carries is also fully derivable from `maker_asset_id` / `taker_asset_id` — exactly one of those is `"0"` (USDC collateral) on every fill, and the side that gave USDC is the buyer. `direction` just makes that derivation explicit.

## Example: simple offset (first page)

```bash
curl "https://api.polynode.dev/v2/onchain/wallets/0xc944d399164b4460a4a79184fc6f419027891e0a/trades?limit=10" \
  -H "x-api-key: YOUR_KEY"
```

## Example: cursor pagination — walk an entire wallet

For wallets with thousands of trades, use cursor mode. Each page is constant-time (~1 second) regardless of depth.

**Page 1 (first page, no cursor yet — pass empty `?cursor=`):**

```bash
WALLET=0xc944d399164b4460a4a79184fc6f419027891e0a
curl "https://api.polynode.dev/v2/onchain/wallets/$WALLET/trades?limit=1000&cursor=" \
  -H "x-api-key: YOUR_KEY"
```

Response (real, captured live) includes a `pagination` block:

      "tx_hash": "0x3611178d615caeab8f1fb32bc6232a32aaf93fc1e2f5bc5c9914ce2ddb9a13c3",

**Page 2+ — echo back the `cursor` from the previous response:**

```bash
curl "https://api.polynode.dev/v2/onchain/wallets/$WALLET/trades?limit=1000&cursor=1777503244:0x3611178d615caeab8f1fb32bc6232a32aaf93fc1e2f5bc5c9914ce2ddb9a13c3_0x2848adef58d655171403ed29b4564cef6d6203a2279149e5759120295724b178" \
  -H "x-api-key: YOUR_KEY"
```

Repeat until `pagination.has_more` is `false`. Done — you've walked the entire wallet's trade history.

**Cursor format:** `<unix_timestamp>:<txHash_orderHash>`. Treat it as opaque — pass it back unchanged. Server-side it filters trades where `(timestamp < lastTs) OR (timestamp == lastTs AND id < lastId)`, so same-timestamp ties (multiple fills in one block) are handled correctly with no dups or skips.
