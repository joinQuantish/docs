Polymarket has deployed a V2 exchange system on Polygon mainnet. polynode detected, decoded, and verified every V2 contract on day one. Our settlement stream, event pipeline, and trading SDK support V2 alongside V1 with zero disruption.

No action is required from existing settlement stream users. V2 trades flow through the same WebSocket feed automatically.

## What We've Verified

All of the following has been tested against live V2 contracts on Polygon mainnet:

- V2 settlement calldata decoded and verified against real on-chain transactions
- EIP-712 order hash computation matches the live V2 exchange's `hashOrder()` function
- V2 OrderFilled and OrdersMatched events decoded from real block receipts
- PolyUSD wrapping and unwrapping executed and detected through our P2P pipeline
- V2 API key derivation (same auth flow as V1, shared database)
- Live V2 order placed and verified on the V2 CLOB orderbook
- Full order lifecycle tested: place, check status, cancel, re-place

## What Changed in V2

### New Exchange Contracts

V2 uses three new exchange contracts. V1 contracts remain active during the transition.

| Contract | Address |
|----------|---------|
| CTF Exchange | `0xe111180000d2663c0091e4f400237545b87b996b` |
| NegRisk Exchange A | `0xe2222d279d744050d28e00520010520000310f59` |
| NegRisk Exchange B | `0xe2222d002000ba0053cef3375333610f64600036` |

### PolyUSD Collateral

V2 uses **PolyUSD** instead of USDC.e as the exchange collateral. PolyUSD is a 1:1 wrapper around USDC.e with 6 decimals.

| Contract | Address |
|----------|---------|
| PolyUSD | `0xc011a7e12a19f7b1f670d46f03b03f3342e82dfb` |
| Collateral Onramp | `0x93070a847efef7f70739046a929d47a521f5b8ee` |
| Collateral Offramp | `0x2957922eb93258b93368531d39facca3b4dc5854` |

To deposit into V2: wrap USDC.e into PolyUSD via the Onramp. To withdraw: unwrap PolyUSD back to USDC.e via the Offramp.

### V2 Order Struct

The V2 order has 11 EIP-712 fields (V1 had 12):

| Field | Type | Notes |
|-------|------|-------|
| salt | uint256 | Unique order identifier |
| maker | address | Order creator's wallet |
| signer | address | Signing key |
| tokenId | uint256 | Conditional token position ID |
| makerAmount | uint256 | Amount maker provides (6 decimals) |
| takerAmount | uint256 | Amount maker receives (6 decimals) |
| side | uint8 | 0 = BUY, 1 = SELL |
| signatureType | uint8 | 0 = EOA, 1 = Proxy, 2 = GnosisSafe |
| timestamp | uint256 | **New.** Order creation time in milliseconds |
| metadata | bytes32 | **New.** Hashed order metadata |
| builder | bytes32 | **New.** Builder attribution code |

**Removed from V1:** `taker`, `expiration`, `nonce`, `feeRateBps`

### EIP-712 Domain

```json
{
  "name": "Polymarket CTF Exchange",
  "version": "2",
  "chainId": 137,
  "verifyingContract": "<exchange_address>"
}
```

## What Didn't Change

- **Token IDs** — same ConditionalTokens contract (`0x4D97DCd97eC945f40cF65F87097ACe5EA0476045`)
- **Market data** — same markets, same condition IDs, same enrichment
- **Authentication** — same API key derivation, same HMAC headers
- **CLOB API paths** — same endpoints (`/order`, `/book`, `/data/orders`)
- **WebSocket protocol** — same subscriptions, same event format
- **Position splits/merges** — same ConditionalTokens events
- **Oracle system** — unchanged

## Settlement Stream (WebSocket)

No changes required for settlement stream subscribers. V2 settlements produce the same event types:

- `settlement` — pending detection from mempool (3-5 seconds pre-confirmation)
- `status_update` — block confirmation with latency measurement
- `trade` — confirmed trade from OrderFilled event
- `deposit` — now includes PolyUSD wrapping/unwrapping alongside USDC.e

V2 detection is automatic. polynode identifies V2 transactions by the `matchOrders` selector (`0x3c2b4399`) and V2 exchange addresses.

### Payload Format

Key formatting rules verified against the live V2 CLOB:

- `salt` — **number** (not string)
- `side` — `"BUY"` or `"SELL"` (**string**, not 0/1)
- `signatureType` — **number** (0, 1, or 2)
- `makerAmount` — **string**, max 4 decimal places in token units
- `takerAmount` — **string**, max 2 decimal places in token units
- `timestamp` — **string**, milliseconds since epoch
- `metadata`, `builder` — **bytes32 hex strings** (zero-padded)
- `owner` — the CLOB API key UUID
- Minimum order size: **5 tokens**

### HMAC Authentication

Same as V1. The HMAC signature uses URL-safe base64:

```
message = timestamp + method + path + body
hmac = HMAC-SHA256(base64_decode(api_secret), message)
signature = base64_encode(hmac)
             .replace('+', '-')
             .replace('/', '_')
```

Headers:
```
POLY_ADDRESS: <wallet_address>
POLY_API_KEY: <api_key>
POLY_SIGNATURE: <url_safe_base64_hmac>
POLY_TIMESTAMP: <unix_seconds>
POLY_PASSPHRASE: <passphrase>
```

## polynode SDK Support

The polynode SDK supports V2 with a single config change:

<CodeGroup>
```rust Rust
let mut config = TraderConfig::default();
config.exchange_version = ExchangeVersion::V2;

let mut trader = PolyNodeTrader::new(config)?;
// Everything else is the same
let result = trader.order(OrderParams {
    token_id: "10293...".into(),
    side: OrderSide::Buy,
    price: 0.05,
    size: 10.0,
    ..Default::default()
}).await?;
```

```typescript TypeScript
const client = new PolyNodeClient({
  exchangeVersion: "v2",
  // ... rest is the same
});

const result = await client.placeOrder({
  tokenId: "10293...",
  side: "BUY",
  price: 0.05,
  size: 10,
});
```

```python Python
client = PolyNodeClient(
    exchange_version="v2",
    # ... rest is the same
)

result = client.place_order(
    token_id="10293...",
    side="BUY",
    price=0.05,
    size=10,
)
```
</CodeGroup>

## Timeline

V2 contracts are deployed on Polygon mainnet and processing test trades. The V2 CLOB is live and accepting orders. Polymarket will announce the full migration date with advance notice. polynode supports both V1 and V2 simultaneously throughout the transition.
